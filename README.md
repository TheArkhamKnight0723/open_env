# ICU Resource Allocation — OpenEnv

An AI environment that simulates critical-care resource allocation during
patient surge scenarios. Designed for the **OpenEnv × Scaler Hackathon**.

The agent plays the role of a charge nurse, deciding which patients to admit
to the ICU and which scarce resources — beds, ventilators, vasopressors, and
nurse hours — to assign under tight constraints and real clinical pressure.

## Quick start

### 1. Install dependencies

```bash
pip install -r requirements.txt
```

### 2. Set environment variables

```bash
export API_BASE_URL="https://api.openai.com/v1"   # or your LLM endpoint
export MODEL_NAME="gpt-4o-mini"
export HF_TOKEN="your-api-key-here"
```

### 3. Run inference (hackathon evaluator entry point)

```bash
python inference.py
```

This runs one episode per task and emits structured `[START]`, `[STEP]`,
and `[END]` log lines on stdout.

### 4. Start the web server locally

```bash
uvicorn app:app --host 0.0.0.0 --port 7860
```

Open `http://localhost:7860` for the Gradio UI, or hit the REST API:

```bash
curl -X POST http://localhost:7860/api/reset \
     -H "Content-Type: application/json" \
     -d '{"task_id": "task_surge_001"}'

curl -X POST http://localhost:7860/api/step \
     -H "Content-Type: application/json" \
     -d '{"action": {"allocations": [...]}}'

curl http://localhost:7860/api/state
```

### 5. Run tests

```bash
python -m pytest tests/ -v
```

## Reward / grading

The grader scores each allocation decision across four components:

| Component | Weight | Description |
|---|---|---|
| `survival_outcome_score` | 0.45 | Expected aggregate survival vs. optimal |
| `resource_efficiency_score` | 0.30 | Correct resource matching, no over-limit |
| `fairness_score` | 0.15 | Severity-driven admissions, not age-driven |
| `triage_correctness_score` | 0.10 | Correct handling of time-sensitive cases |

Final reward is clamped to **[0.0, 1.0]**.

## Agents

**Rule-Based Agent** — Greedy heuristic that sorts patients by a composite
priority score (`severity × 2 + time_sensitive × 1.5 + prognosis`) and
admits in order until a resource cap is reached. Deterministic and fast.

**LLM Agent** — Sends the current observation as a structured prompt to an
OpenAI-compatible endpoint and parses the JSON allocation it returns.
Falls back to the rule-based agent on any error.

**RL Agent (PPO)** — Trained via stable-baselines3 on the ICUGymWrapper.
Train it with:

```bash
python -m training.train_ppo --timesteps 100000 --seed 42
```

Compare all agents:

```bash
python -m training.evaluate --agent all --episodes 5
```
